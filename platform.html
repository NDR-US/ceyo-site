PLATFORM · ARCHITECTURE · NON-CUSTODIAL SEALING

A Sidecar Evidentiary Layer
For Inference Systems

CEYO attaches at the inference boundary (sidecar, wrapper, or gateway) to capture policy-scoped decision-state context, canonicalize it deterministically, and seal it cryptographically for independent verification under constrained disclosure.

Architecture Summary
CEYO produces a canonical artifact (stable structure + stable bytes) and applies cryptographic sealing (hash + signature). Policies define scope and disclosure tiers before generation. Signing keys remain in the operator environment (HSM/KMS/TEE).
01 / Capture Policy-scoped
Inclusion, exclusion, masking, tiering — defined pre-generation. Policy version is embedded in every artifact.
02 / Canonicalize Deterministic
Stable schema + stable byte output enabling independent recomputation of integrity checks.
03 / Seal Tamper-evident
Hashing establishes tamper-evidence; signatures establish origin authenticity.
04 / Verify Independent
Verification confirms integrity and provenance under policy scope, not correctness of output.
REFERENCE FLOW

From Inference Event
To Sealed Artifact

CEYO’s flow is intentionally narrow and repeatable: capture declared fields, normalize into a canonical artifact, seal with cryptography, and enable verification without forcing disclosure.

Implementation specifics vary by deployment pattern, but the evidentiary pipeline remains stable across environments.

Design Requirements
  • Policy-defined scope before generation
  • Deterministic canonicalization for reproducible bytes
  • Non-custodial key boundary (operator retains private keys)
  • Fail-open availability (inference continuity preserved)

ARTIFACT SHAPE

Canonical Artifact Header
Minimal Example

Public pages are descriptive. A controlled review includes a reference schema, canonicalization procedure, and verification steps. The artifact must make scope explicit: policy identifier, version, and disclosure tier.

Do not over-collect. CEYO’s strength is declared minimization with deterministic integrity — not exhaustive surveillance.

Boundary Note
Verification confirms integrity and provenance under policy scope — not correctness, legality, fairness, or compliance outcomes.

DEPLOYMENT PATTERNS

Attach Without Mutating
Inference Authority

A / Sidecar Near runtime
Deploy alongside the inference service. Capture and seal per request without changing model outputs. Best for service-based inference.
B / Gateway Edge boundary
Wrap calls at an ingress/egress boundary. Useful when multiple services share a policy and disclosure tiering is centralized.
C / Wrapper Library
Instrument at the SDK level. Useful for internal tooling and research environments where policy is embedded in code.
