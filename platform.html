Architecture
Beside Inference

CEYO is deployed at the inference boundary as a sidecar, wrapper, gateway, or service mesh component. It captures policy-declared fields, normalizes to a stable artifact schema, canonicalizes deterministically, and seals cryptographically using client-controlled keys.

01 / Deployment Sidecar / Wrapper / Gateway
CEYO can run as a co-located sidecar, integrated wrapper library, edge gateway, or mesh filter. The operational goal is consistent: capture policy-defined context at inference-time without becoming the decision engine.
02 / Pipeline Normalize → Canon → Seal
The artifact pipeline is deterministic by design:
1) Capture (policy scope) · 2) Normalize (stable schema) · 3) Canonicalize (stable bytes) · 4) Hash · 5) Sign · 6) Store/Transmit
03 / Disclosure Public / Redacted / Controlled
Disclosure tiers are defined pre-generation by policy. Artifacts can be emitted as public summaries, redacted records, or controlled-review bundles to support oversight without default exposure of proprietary internals.
04 / Availability Fail-open
Inference continuity remains independent of sealing. If artifact generation fails, inference proceeds; the event is logged and the record is marked unsealed for review.
DEPLOYMENT MODES

Integrate Without
Becoming The Model

CEYO is meant to attach beside existing inference systems. It does not require model weight custody and does not require architectural disclosure beyond policy scope.

Mode A Sidecar
Co-located process captures request/response context and runtime signals declared by policy. Strong fit for Kubernetes and service mesh environments.
Mode B Wrapper SDK
Library-level integration around inference calls (client or server). Lowest latency overhead, best for embedded or constrained deployments.
Mode C Gateway
Centralized boundary enforcing policy capture and tiered disclosure across multiple inference services. Strong fit for regulated environments with centralized logging controls.
ARTIFACT PIPELINE

From Context
To Verifiable Record

01 · Capture
Policy-scoped
Capture is declared before generation: inclusion, exclusion, masking, tier rules, and policy version identifiers embedded into the artifact.
02 · Normalize
Stable schema
Captured fields are mapped into a deterministic artifact schema so verification can be repeated across environments without interpretive logging variance.
03 · Canonicalize
Stable bytes
Canonicalization produces deterministic byte output for hashing and signature verification, independent of JSON field order or serialization differences.
04 · Seal
Hash + signature
Hashing establishes tamper-evidence; signatures establish origin authenticity (suite configurable). Verification becomes independent and reproducible.
05 · Keys
Non-custodial
Signing keys remain in client-controlled HSM/KMS/TEE. CEYO does not store, transmit, or possess private keys.
06 · Verify
Independent recomputation
Oversight can recompute canonical bytes, validate hashes, and validate signatures under declared scope. Verification proves integrity—nothing more.
