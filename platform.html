Platform & Architecture

Decoupled
Verification Architecture

CEYO is designed as an external evidentiary substrate that attaches at the inference boundary and produces deterministic, policy-scoped artifacts — canonicalized and sealed for later institutional verification under constrained disclosure.

Attachment Patterns

Inference Boundary Integration

CEYO integrates at the inference boundary through one of three operator-selected configurations. The inference engine remains unchanged. The evidentiary layer operates externally.

Attachment is designed to minimize operational friction while preserving system integrity and institutional separations.

Non-interference affirmations
× No model weight access or mutation
× No intervention in inference decisions
× No operational control assumption
→ Record generation only
01 / Sidecar Co-located process
Runs alongside the inference service (container/pod). Observes defined inference events, captures policy-scoped fields, and generates artifacts without mutating request/response payloads.
02 / Gateway Boundary proxy
Positioned between caller and inference endpoint. Suitable for API inference architectures. Fail-open enforcement can be configured at the proxy layer.
03 / Wrapper Interface-compatible
Wraps inference calls while preserving an expected interface. Useful for environments requiring compatibility guarantees and explicit routing control.
Capture Policy

Define Before You Collect

Artifact generation begins with explicit capture policy definition. Policy determines which fields are included, excluded, and masked, as well as disclosure tier configuration.

Policy identifiers and versions are embedded in every artifact, making capture boundaries reviewable under institutional scrutiny.

01 / Fields Inclusion + exclusion
Policy defines the capture surface: included contextual fields, explicit exclusions (e.g., weights, training datasets), and masking rules for sensitive elements.
02 / Versioning Reviewable scope
Policy ID, policy version, and (where configured) a policy hash are recorded in the artifact to prevent post hoc scope drift and to support alignment checks during verification.
03 / Disclosure Tiered visibility
Public/Redacted and Controlled Review tiers allow integrity validation without default exposure of proprietary internals. Tier boundaries are defined before capture.
Deterministic Canonicalization

Stable Bytes Across Environments

CEYO treats canonicalization as a structural requirement. Without deterministic canonicalization, verification becomes environment-dependent and disputes revert to interpretation.

Canonicalization is applied after policy-defined masking and exclusion. Integrity is computed over the post-redaction canonical form.

RFC 8785 JSON canonicalization
RFC 8785 defines deterministic JSON serialization rules (ordering, numeric normalization, encoding) to ensure logically equivalent structures yield identical byte streams.
Ordering Non-negotiable
Redaction → Canonicalization → Deterministic byte stream → Hash → Signature. Any deviation weakens reproducibility of integrity checks.
Testability Fixed fixtures
Determinism must be unit-testable using fixed fixtures such that the same input + policy version produces identical canonical bytes and identical digests across environments.
Integrity Sealing

Non-Custodial Cryptographic Posture

After canonicalization, artifacts are sealed using cryptographic hashing and digital signatures. Reference implementations include SHA-256 and ECDSA (P-256), with suite selection configurable by institutional requirements.

Private signing keys remain within operator-controlled infrastructure (HSM/KMS/TEE). CEYO does not possess, transmit, or store private signing keys.

01 / Hashing Tamper-evidence
Compute a cryptographic digest over canonical bytes. Any post-seal alteration produces a different digest, enabling mechanical detection of modification.
02 / Signature Origin authenticity
Sign the digest under operator key custody. Signature validation confirms origin attribution under the declared public-key material and custody framework.
03 / Availability Fail-open
Inference continuity remains independent of sealing. If sealing fails, inference proceeds. Failures are logged and artifacts are marked unsealed and reviewable.
