Trust in CEYO is not a claim. It is a procedure: deterministic canonicalization, cryptographic sealing, and independent verification under policy scope. The mechanism is designed to hold up under scrutiny.

Trust Summary
CEYO’s trust posture is implemented through: (1) explicit capture scope, (2) deterministic canonicalization, (3) cryptographic sealing, (4) non-custodial key boundaries, and (5) tiered disclosure.
01 / Determinism Stable bytes
Open
Canonicalization produces a stable byte representation independent of field order and runtime differences. Independent parties can recompute the same hash from the same declared inputs.
02 / Sealing Hash + signature
Toggle
Hashing establishes tamper-evidence. Signatures establish origin authenticity. Algorithm suites can be selected to match institutional requirements. The verification procedure is explicit.
03 / Key Custody Non-custodial
Toggle
Private signing keys remain in the operator-controlled environment (HSM/KMS/TEE). CEYO does not possess, transmit, or store private keys. Authority remains with institutions and operators.
VERIFICATION MECHANICS

What An Auditor Verifies
In Practice

A verifier does not need model weights to verify an artifact’s integrity. They need: the artifact, the declared policy scope, the canonicalization procedure, and the signing public key material (as authorized).

Verification is reproducible: canonicalize → hash → verify signature → confirm policy identifiers and disclosure tier.

Verification Checklist
  • Policy ID + version embedded and matches declared scope
  • Canonicalization algorithm is specified (e.g., RFC8785)
  • Hash recomputation matches recorded digest
  • Signature validates under authorized public key
  • Disclosure tier is explicit (Public / Redacted / Controlled)
DISCLOSURE TIERS

Constrained Disclosure
By Design

Public Descriptive
Open
Public material describes architecture and boundaries. It does not include sensitive operational details or private key material.
Redacted Minimized
Toggle
Redacted artifacts exclude or mask fields per policy. Integrity remains verifiable for the disclosed structure under declared scope.
Controlled Review Authorized
Toggle
Controlled review is a process. Technical specifics, schemas, and verification materials are shared under authorization and terms.
OPERATIONAL POSTURE

Fail-Open Availability
Inference Continues

Evidence generation must not become a single point of failure for inference. If sealing fails, inference proceeds. Failure events are logged; artifacts are marked accordingly.

This preserves operational continuity while still making integrity posture reviewable.
